## Слайд 4. Актуальность темы

**Содержимое:**

Актуальность работы обусловлена следующими факторами:

- обращения граждан поступают из нескольких каналов и имеют разнородную структуру;
- оператору необходим единый визуальный интерфейс для ежедневной работы;
- без качественного frontend-слоя даже корректно работающий backend не обеспечивает эффективное использование системы;
- современная платформа должна быть адаптивной, понятной, устойчивой к ошибкам и доступной для разных категорий пользователей.

**Визуал:** схема «разрозненные каналы → единый интерфейс оператора».

## Слайд 5. О проекте и роли frontend-части

**Содержимое:**

Разрабатываемый проект представляет собой **fullstack-веб-приложение для управления городскими проблемами**.

В общей архитектуре системы:

- **backend** выполняет сбор данных из внешних источников, обработку и NLP-классификацию;
- **frontend** предоставляет оператору рабочее пространство для взаимодействия с результатами этой обработки.

На текущем этапе frontend реализует:

- дашборд жалоб;
- карточку отдельной жалобы;
- управление источниками ВКонтакте и email;
- управление метками;
- формирование PDF-отчётов;
- центр уведомлений и пользовательские элементы интерфейса.

**Визуал:** блок-схема  
`Источники → Backend/NLP → Frontend-дашборд → Оператор`.

## Слайд 9. Архитектура frontend-приложения

**Содержимое:**

Frontend построен по модульному принципу.

**Структура решения:**
- `app/` — маршруты и страницы Next.js App Router;
- `components/` — UI, сущности, фичи, виджеты;
- `lib/api/` — классы для работы с REST API;
- `lib/hooks/` — пользовательские React hooks;
- `lib/types/` — типы TypeScript;
- `lib/utils/` — утилиты и преобразование данных.

**Архитектурные особенности:**
- серверное состояние — через TanStack Query;
- HTTP-взаимодействие — через ky;
- real-time обновления — через socket.io-client;
- UI-слой — shadcn/ui + Tailwind CSS;
- приложение собирается в standalone-режиме.

**Визуал:** слоистая схема  
`UI → Hooks → API-классы → Backend REST/WebSocket`.

## Слайд 12. Упрощённая диаграмма классов frontend

**Содержимое:**

Для frontend-части применима упрощённая диаграмма классов/модулей:

- `ComplaintPage` использует `ComplaintStatsBlock` и `ComplaintDataTable`
- `ComplaintDataTable` использует:
  - `useComplaints`
  - `useComplaintsSocket`
  - `useLabels`
- `ComplaintContent` использует:
  - `useUpdateComplaint`
  - `useUpdateComplaintLabels`
- `useComplaints` работает через `ComplaintAPI`
- `useSourceManagement` работает через `VkApi` и `EmailApi`
- `ReportPage` использует `ComplaintAPI` и `ComplaintsAdminReportDocument`

**Визуал:** схема связей модулей, а не перегруженная UML.

## Слайд 15. Заключение и перспективы развития

**Содержимое:**

В результате работы была разработана frontend-часть платформы для управления городскими проблемами, обеспечивающая:

- визуализацию обращений;
- управление статусами и метками;
- настройку источников;
- получение уведомлений;
- формирование отчётности.

**Перспективы развития:**
- расширение accessibility (a11y);
- локализация интерфейса;
- публичный гостевой режим по трек-номеру;
- расширение поддержки устройств и браузеров;
- PWA-режим;
- дальнейшее развитие интерфейсов аналитики и ролей пользователей.

**Финальная мысль для защиты:**
Frontend в данном проекте выступает не просто как «визуальная оболочка», а как ключевой инструмент практического использования результатов backend- и NLP-обработки.
